# Application Intelligence Engine — Approved Design

## Source of truth
The Application Intelligence Manual (AIM) at `attached_assets/VisaPrep™_Application_Intelligence_Manual_(AIM)_1784228263182.pdf` governs all engine behavior. Do not reinterpret or simplify its framework without owner approval.

## Approved decisions (owner-confirmed)
- **Submission date**: extract from PDF only if explicitly present; never infer from other dates; omit time-based insights if absent.
- **firstName**: extract given name from DS-160 name fields; return null if not identifiable. Used for personalized encouragement only.
- **Missing sections**: set keySignals to [] and insights to [{ observation: "Not enough information…", whyItMatters: "", preparationGuidance: "" }].
- **Document validation**: model assesses isLikelyDS160 before generating analysis; invalid docs return empty arrays and a user-facing message without fake analysis. firstName set to null on invalid doc.
- **Session key**: `visaprep_analysis_v3` — bumped from v2 when firstName was added to schema.
- **Application Profile**: emphasize the application (visa type, purpose, employment, travel plans), not the person.
- **Strengths-first prompt**: system prompt enforces genuine strengths before preparation areas; prohibits alarmist language.
- **Free/paid boundary (server-enforced)**: `sections`, `crossSectionObservations`, `readyToExplain` are generated by the AI but stripped from the `/api/analyze` response before returning to the client. Free users cannot access paid fields even via the API JSON. Full analysis must be stored server-side and delivered only after payment verification (Task #8).

## Output schema (current — v3, AI generates all fields)
```json
{
  "documentAssessment": { "isLikelyDS160": true, "message": null },
  "applicationProfile": "string",
  "submissionDate": "string | null",
  "firstName": "string | null",
  "strengths": [{ "label": "string", "detail": "string" }],
  "topPreparationAreas": [
    { "title": "string", "observation": "string", "whyItMayComeUp": "string", "whatToBeReadyToExplain": "string" }
  ],
  "sections": [ /* paid — stripped from free response */
    {
      "lesson": "string",
      "interviewWeight": "string",
      "memoryRisk": "string",
      "keySignals": ["string"],
      "insights": [{ "observation": "string", "whyItMatters": "string", "preparationGuidance": "string" }]
    }
  ],
  "crossSectionObservations": [ /* paid — stripped from free response */
    { "title": "string", "connection": "string", "whyItMatters": "string", "whatToReview": "string" }
  ],
  "readyToExplain": [ /* paid — stripped from free response */
    { "topic": "string", "whyItMayComeUp": "string", "whatToBeReadyToExplain": "string", "possibleQuestions": ["string"] }
  ]
}
```
Sections always appear in AIM lesson order 1→7; all seven always present.

## Free frontend layout (current — v3)
1. File Analyzed header — green checkmark + filename + "Try another file" link
2. Applicant Snapshot — applicationProfile + submissionDate
3. Personalized encouragement — uses firstName ("Ada, you're off to a good start.") with null fallback
4. Strengths in Your Application — strengths array with green checkmarks; disclaimer footnote
5. Application Highlights — topPreparationAreas: title + observation + "Why this may come up" only (whatToBeReadyToExplain NOT shown in free tier)
6. Premium CTA — "Start My Personalized Interview Preparation" → payment modal
7. Invalid doc state: amber card "We Need Your DS-160" with Upload another button

**Removed from free tier (product refinement sprint):** "What You Should Be Ready to Explain" section, "View Full Analysis" collapsible, crossSectionObservations, sections/LessonCards.
